1. Основной процесс расчета прогноза
Название: Прогнозирование медицинских расходов пациента для назначения страховой премии

Участники (объекты):
Аналитик – пользователь.
Веб-интерфейс (Frontend) – форма ввода, кнопки.
Система (Backend + ML) – обработка данных, модель, выдача результата.
База данных (БД) – хранение результатов.

Описание потока:

1. Аналитик открывает вкладку "Новый прогноз".
2. Веб-интерфейс загружает форму.
3. Аналитик заполняет поля (возраст, пол, ИМТ и т.д.) и нажимает кнопку "Рассчитать".
4. Веб-интерфейс отправляет данные (JSON) в Систему (Backend API).
5. Система (Backend) валидирует данные, обогащает их и вызывает ML-модель для инференса.
6. ML-модель возвращает рассчитанный прогноз (число).
7. Система (Backend) формирует ответ и отправляет его обратно во Веб-интерфейс.
8. Веб-интерфейс отображает результат на экране аналитика.
9. Одновременно Система (Backend) отправляет команду на сохранение (INSERT) в Базу данных.
10. База данных подтверждает сохранение записи.

2. Получение факторов риска
Название: Просмотр топ-3 факторов риска

Участники (объекты):
Аналитик 
Веб-интерфейс (Frontend)
Система (Backend + ML + SHAP)
База данных (БД) - сохранения топ-3 фактора в БД

Описание потока:
1. Аналитик получил результат расчёта (видит сумму 245 000 ₽).
2. Аналитик нажимает на кнопку "Факторы риска".
3. Веб-интерфейс отправляет запрос к Backend на получение объяснения (SHAP values) для данного пациента.
4. SHAP-модуль возвращает список вкладов каждого признака (например: "Возраст: +45 000", "Диабет: +38 000").
5. Backend формирует ответ на Frontend.
6. Веб-интерфейс отображает их аналитику (обычно в виде списка или водопадной диаграммы).
7. Одновременно Backend отправляет команду на сохранение (INSERT) в Базу данных.
8. База данных подтверждает сохранение записи.


3. История и поиск
Название: Просмотр истории прогнозов и поиск по объектам

Участники (объекты):
Аналитик
Веб-интерфейс (Frontend)
Система (Backend API)
База данных (БД) – хранилище истории.

Описание потока:

1. Аналитик нажимает на кнопку "История" в меню.
2. Веб-интерфейс отправляет GET-запрос в Систему на получение списка (обычно с параметром парметром пагинации).
3. Система обращается к БД.
4. БД возвращает набор записей (список пациентов с их параметрами, суммой и датой).
5. Система возвращает JSON-массив во Веб-интерфейс.
6. Веб-интерфейс отображает таблицу (ФИО, Возраст, Сумма, Дата).
7. Аналитик вводит текст в строку поиска (например, фамилию "Иванов" или его ID) и нажимает Enter.
8. Веб-интерфейс отправляет GET /history?search=Иванов в Систему.
9. Система выполняет SQL-запрос с условием WHERE name LIKE '%Иванов%'.
10. БД возвращает отфильтрованные данные.
11. Система возвращает их на Веб-интерфейс.
12. Веб-интерфейс перерисовывает таблицу, показывая только найденные записи.

4. Формирование отчёта в виде PDF-файла
Название: Скачать отчет по пациенту

Краткое описание: Аналитик может сохранить результат прогноза в виде красиво оформленного PDF-файла, чтобы отправить клиенту или приложить к страховому делу.

Участники (объекты):
Аналитик
Веб-интерфейс (Frontend)
Система (Backend API)

Описание потока:

1. Аналитик получил прогноз на экране.
2. Система показывает кнопку «Экспорт в PDF».
3. Аналитик нажимает на кнопку «Экспорт в PDF».
4. Frontend отправляет POST-запрос на /export/pdf с prediction_id текущего прогноза.
5. Backend генерирует PDF (данные пациента, прогноз, доверительный интервал, топ-3 фактора, дату, логотип компании).
6. Backend возвращает PDF-файл (Content-Type: application/pdf).
7. Браузер запускает скачивание файла прогноз_{фамилия}_{дата}.pdf.
8. Аналитик открывает PDF.	

Файл содержит: данные пациента, прогноз, доверительный интервал, топ-3 фактора, дату, логотип компании
Альтернативный поток: PDF можно сгенерировать не только для текущего прогноза, но и для любого сохраненного из истории (кнопка «Экспорт» в строке таблицы).

5. Удаление прогноза из истории
Название: Удалить ошибочный или тестовый прогноз

Аналитик может удалить прогноз из истории, если он был сделан по ошибке, для теста или содержит некорректные данные.

Участники (объекты):
Аналитик
Веб-интерфейс (Frontend)
Система (Backend API)
База данных (БД) – хранилище истории.

Описание потока:
1. Аналитик Открывает раздел «История».
2. Система показывает таблицу прогнозов.
3. Аналитик находит строку с ошибочным прогнозом, в каждой строке есть кнопка/иконка 🗑️ (корзина).
4. Аналитик нажимает на иконку корзины.
5. Frontend показывает модальное окно: «Удалить прогноз для пациента Иванов И.?» с кнопками [Да] / [Нет].
6. Аналитик нажимает [Да].
7. Frontend отправляет DELETE-запрос на /history/{prediction_id}.
8. Backend удаляет запись из БД (SQL DELETE).
9. БД подтверждает удаление.
10. Backend возвращает статус 200 OK.
11. Система показывает уведомление: «Прогноз удален» Таблица обновляется (строка исчезает).
12. Альтернативный поток: Аналитик нажимает [Нет] → модальное окно закрывается, удаление не происходит.

6. Сортировка в истории

Название: Отсортировать прогнозы по дате, сумме или возрасту

Аналитик может кликнуть на заголовок столбца в таблице истории, чтобы отсортировать записи (например, показать сначала самые дорогие прогнозы или самые свежие).

Участники (объекты):
Аналитик
Веб-интерфейс (Frontend)
Система (Backend API)

Описание потока:

1. Аналитик смотрит на таблицу истории, заголовки столбцов: Дата, Пациент, Возраст, Прогноз, Решение
2. Аналитик нажимает на заголовок «Прогноз»
3. Frontend сортирует уже загруженный массив данных по полю predicted_cost
4. Рядом с заголовком появляется иконка ▼, система перерисовывает таблицу
5. Аналитик нажимает на заголовок «Прогноз» еще раз
6. Процесс повторяется, иконка меняется на ▲, таблица показывает сначала самые дешевые

7. Фильтры во вкладке История

Название: Отфильтровать прогнозы по стоимости прогноза и дате

Участники (объекты):
Аналитик
Веб-интерфейс (Frontend)
Система (Backend API)
База данных (БД)

Описание потока:

1. Аналитик открывает раздел "История" -> "Фильтры".
2. Аналитик вводит минимальное значение прогноза в ячейку "Прогноз от".
3. Frontend фильтрует загруженный массив данных по полю predicted_cost и отображает только те строки, где "Прогноз" больше или равен введённому значению.
4. Аналитик вводит максимальное значение прогноза в ячейку "Прогноз до".
5. Frontend фильтрует массив данных по полю predicted_cost и отображает только те строки, где "Прогноз" меньше или равен введённому значению.
6. Аналитик вводит раннюю дату прогноза в ячейку слева в формате дата.
7. Frontend фильтрует массив данных по полю created_at и отображает только те строки, где "Дата" больше или равна введённой даты.
8. Аналитик вводит позднюю дату прогноза в ячейку справа в формате дата.
9. Frontend фильтрует массив данных по полю created_at и отображает только те строки, где "Дата" меньше или равна введённой даты.
10. Аналитик нажимает на "Сбросить фильтры".
11. Frontend восстанавливает исходный массив и очищает поля ввода.

8. Загрузка анкеты

Название: Распознать анкету, загрузить данные и выполнить прогноз.

Участники (объекты):
Аналитик
Веб-интерфейс (Frontend)
Система (Backend API)
OCR

Описание потока:

1. Аналитик открывает вкладку "Новый прогноз".
2. Аналитик нажимает на кнопку "Распознать анкету".
3. Frontend открывает диалог выбора файла из локального хранилища.
4. Аналитик выбирает файл для загрузки.
5. Frontend загружает файл и отправляет на Backend.
6. Backend с помощью OCR распознаёт данные и формирует массив данных для Frontend.
7. Backend возвращает JSON с распознанными полями.
8. Frontend заполняет поля анкеты полученными данными.
9. Аналитик проверяет и при необходимости корректирует заполненные поля.
10. Аналитик нажимает "Рассчитать".
