Спецификация требований к ПО для MedCost Prediction App
1. Введение
    
    1.1 Цель
        Данный документ описывает функциональные и нефункциональные требования к прототипу системы «MedCost Prediction App». 
        Система предназначена для прогнозирования годовых медицинских расходов пациента на основе введённых аналитиком данных. 
        Спецификация создана для команды разработки с целью унификации понимания требований к продукту.

    1.2 Соглашения, принятые в документе
        Все функциональные требования (ФТ) имеют идентификатор в формате ФТ-X.Y.Z, где X — номер группы функций, Y — номер подгруппы, Z — порядковый номер требования. 
        Уникальный код требования позволяет однозначно идентифицировать его при внесении изменений или создании перекрёстных ссылок. 
        Приоритет всех требований на текущем этапе разработки — «Высокий».

    1.3 Границы проекта
        Прототип системы «MedCost Prediction App» — это hi-fidelity прототип, который позволяет страховому аналитику ввести данные пациента, 
        получить прогноз ожидаемых медицинских расходов, а также проанализировать ключевые факторы риска. 
        Продукт не выполняет интеграцию с внешними учетными системами страховой компании и, на данный момент, не предназначен для конечных потребителей.

    1.4 Ссылки
        Репозиторий проекта: https://github.com/horacemtb/medcost-prediction-app
        Датасет для обучения ML-модели: https://www.kaggle.com/datasets/miadul/medical-cost-predication-dataset/data

2. Общее описание
    2.1 Общий взгляд на продукт
        «MedCost Prediction App» — это новый веб-ориентированный продукт, призванный заменить субъективную ручную оценку страховых рисков. Система является автономной и включает в себя:
        ML-модель: для расчёта прогноза на основе введённых данных.
        Backend: для бизнес-логики, взаимодействия с ML-моделью и базой данных.
        Frontend: для удобного ввода данных, отображения результатов и навигации.
        База данных: для сохранения пациентов и истории расчётов.

        Ключевая бизнес-ценность - сокращение времени андеррайтинга с 2 часов до 10 минут и повышение обоснованности тарифов страхования.

    2.2 Классы пользователей и их характеристики
        Страховой аналитик: Основной пользователь системы, который взаимодействует со всеми её функциями: ввод данных, запуск расчёта, просмотр результатов, факторов риска и истории.
        Основные KPI: коэффициент убыточности, скорость обработки заявки, обоснованность тарифа.
        Инструменты в работе: Excel, BI-системы, внутренние корпоративные сервисы.
        Роли в системе: Полный доступ ко всем функциям (ввод, расчёт, просмотр, удаление, экспорт).

    2.3 Операционная среда
        Тип приложения: Веб-приложение.
        Платформа: Кроссплатформенное (Windows, macOS, Linux).
        Развертывание: Локально с использованием Docker.

    2.4 Ограничения проектирования и реализации
        Языки программирования: Backend — Python, Frontend — React.
        База данных: PostgreSQL.
        Контейнеризация: Использование Docker для развертывания.
        Требования к данным: Использование синтетического датасета, 3 нормальная форма БД.

    2.5 Предположения и зависимости
        Сетевая доступность: 
            Предполагается, что все компоненты системы (Frontend, Backend, База данных) 
            развернуты в одной сетевой среде (например, на локальной машине) и имеют возможность взаимодействовать друг с другом по протоколу HTTP.

        Доступность портов: 
            Предполагается, что порт 8501 (для Frontend) и 8000 (для FastAPI) свободны на машине, 
            где развертывается приложение, если не используются кастомные настройки.

        Зависимости: 
            Для корректной работы Frontend-клиента требуется, чтобы в переменной окружения API_BASE_URL (или в коде по умолчанию) 
            был указан корректный сетевой адрес Backend-сервера.

3. Функции системы
    3.1 Прогнозирование медицинских расходов
        3.1.1 Описание
            Пользователь заполняет форму с данными пациента, система на основе этих данных вычисляет прогнозируемую стоимость медицинских 
            расходов на следующий год. Для обеспечения гибкости и полноты данных, пользователю доступны два способа заполнения формы: 
            автоматическое распознавание данных из загруженного файла (OCR) и ручное заполнение формы.

        3.1.2 Use Case
            Вариант использования: Прогнозирование годовых медицинских расходов пациента
            
            Контекст использования: Страховой аналитик рассчитывает ожидаемую стоимость медицинского обслуживания для назначения страховой премии.
            
            Область действия: Подсистема «Расчёт прогноза» (Frontend + Backend + ML + БД)
            
            Уровень: Цель пользователя (высокоуровневый)
            
            Основное действующее лицо: Страховой аналитик (Андеррайтер)
            
            Участники (дополнительные лица): Веб-интерфейс (Frontend), Backend API, ML-модель, База данных
            
            User story: «Как страховой аналитик, я хочу ввести данные о пациенте (вручную или через загрузку скан-анкеты) 
            и получить прогноз расходов, чтобы быстро и обоснованно рассчитать страховую премию».
            
            Предусловие: Аналитик авторизован в системе (если требуется). Backend и ML-модель доступны, БД подключена.
            
            Минимальные гарантии успеха: Если операция прервана, система не сохраняет неполные данные. Входные данные не теряются в процессе передачи (валидация на клиенте).
            
            Гарантии успеха: При успешном завершении: прогноз рассчитан, результат отображён пользователю, запись сохранена в БД вместе с входными параметрами.
            
            Триггер: Аналитик нажимает кнопку «Рассчитать прогноз» после заполнения формы (или загрузки файла).
            
            Описание (основной сценарий): 
                1. Аналитик выбирает способ ввода: ручной или OCR. 
                2. Если ручной — вводит данные в поля формы. 
                3. Если OCR — загружает скан/фото анкеты; система распознаёт текст и заполняет поля автоматически. 
                4. Аналитик проверяет/корректирует данные, нажимает «Рассчитать». 
                5. Frontend отправляет JSON на POST /api/predict. 
                6. Backend валидирует данные, вызывает ML-модель. 
                7. Модель возвращает predicted_cost. 
                8. Backend сохраняет запись в БД (таблица predictions), получает prediction_id. 
                9. Backend возвращает {prediction_id, predicted_cost} на Frontend. 
                10. Frontend отображает прогноз и оценку в процентиле. 
                11. Пользователь видит кнопку для просмотра факторов риска.
            
            Расширения (альтернативные сценарии): 
                3a. OCR не распознал текст → уведомление об ошибке, предложение ввести данные вручную. 
                5a. Невалидные данные (возраст >100) → Backend возвращает ошибку 400, Frontend подсвечивает поле. 
                6a. ML-модель недоступна → Backend возвращает ошибку 503, Frontend предлагает повторить позже. 
                8a. Ошибка сохранения в БД → Backend логирует ошибку, но прогноз всё равно возвращается (без истории).

        3.1.3 Функциональные требования
            ФТ-3.1.1 (Ввод данных): 
                Система должна предоставлять веб-форму для ввода следующих параметров пациента, согласно схеме PredictionInput:

                    full_name: строковое поле (от 2 до 255 символов).
                    age: целое число (от 18 до 100).
                    gender: флажок/список (Женский (0), Мужской (1)).
                    bmi: число с плавающей точкой (от 10 до 60).
                    smoker, diabetes, hypertension, heart_disease, asthma: логические поля (Да/Нет).
                    physical_activity_level: список ("Low", "Medium", "High").
                    daily_steps: целое число (от 0 до 50000).
                    sleep_hours: число с плавающей точкой (от 0 до 24).
                    stress_level: целое число (от 1 до 10).
                    doctor_visits_per_year: целое число (от 0 до 100).
                    hospital_admissions: целое число (от 0 до 50).
                    medication_count: целое число (от 0 до 100).
                    city_type: список ("Urban", "Semi-Urban", "Rural").
                    previous_year_cost: число с плавающей точкой (от 0).
                    snils:целое число (от 18 до 100). Обязательное поле. 
                    phone:целое число (от 18 до 100).
                    address:строковое поле (от 2 до 255 символов).

            ФТ-3.1.2 (OCR-заполнение формы): 
                Система должна предоставлять возможность загрузить файл с данными пациента (скан, фото). 
                После успешного распознавания система автоматически заполняет соответствующие поля веб-формы на основе извлеченных данных.

            ФТ-3.1.3 (Отправка данных): 
                При нажатии пользователем кнопки "Рассчитать", система должна отправить данные в формате JSON на API-эндпоинт 
                POST /api/predict

            ФТ-3.1.4 (Расчет прогноза): 
                Backend-сервис, получив корректные данные, должен выполнить инференс ML-модели и вернуть JSON-ответ 
                с рассчитанной прогнозируемой стоимостью медицинских расходов на следующий год (predicted_cost) 
                и уникальным идентификатором прогноза (prediction_id).

            ФТ-3.1.5 (Отображение результата): 
                После получения прогноза, Frontend должен отобразить пользователю:
                    Рассчитанную сумму (predicted_cost).
                    Оценку в процентиле относительно распределения расходов в датасете.
                    Кнопку для просмотра деталей и факторов риска.

            ФТ-3.1.6 (Сохранение истории): 
                Каждый успешно выполненный прогноз должен автоматически сохраняться в базе данных вместе со всеми 
                входными параметрами и результатом, как запись таблицы predictions в схеме PredictionRecord.

    3.2 Получение аналитики по факторам риска
        3.2.1 Описание
            Пользователь может запросить детальный анализ факторов, оказавших наибольшее влияние на сформированный прогноз. 
            Система анализирует вклад каждого признака и представляет его в понятном виде.

        3.2.2 Use Case
            Вариант использования: Просмотр топ факторов, повлиявших на прогноз

            Контекст использования: Аналитику нужно объяснить клиенту или руководству, почему прогноз получился высоким (или низким).

            Область действия: Подсистема «Аналитика рисков» (Frontend + Backend + SHAP + БД)

            Уровень: Цель пользователя

            Основное действующее лицо: Страховой аналитик

            Участники: Frontend, Backend API, SHAP-модуль, База данных

            User story: «Как аналитик, я хочу видеть, какие именно признаки (курение, возраст, диабет) больше всего увеличили прогноз, 
            чтобы обосновать решение клиенту».

            Предусловие: Прогноз для данного prediction_id уже рассчитан и сохранён в БД.

            Минимальные гарантии успеха: Если запрос факторов не удался, показывается общая ошибка, но основной прогноз остаётся на экране.

            Гарантии успеха: Пользователь получает список факторов, отсортированных по влиянию (SHAP-значение), с указанием признака,
            его значения и вклада в прогноз.

            Триггер: Аналитик нажимает кнопку «Показать факторы риска» для текущего или сохранённого прогноза.

            Описание (основной сценарий): 
                1. Аналитик находится на экране результата прогноза (с prediction_id). 
                2. Нажимает «Факторы риска». 
                3. Frontend отправляет GET /api/predictions/{prediction_id}/factors. 
                4. Backend проверяет, есть ли уже рассчитанные факторы в БД. 
                5. Если нет — вызывает SHAP-модуль для этого набора признаков, вычисляет вклады. 
                6. Backend сохраняет факторы в таблицу risk_factors. 
                7. Backend возвращает массив {feature_name, feature_value, shap_value, rank}. 
                8. Frontend отображает их (обычно список или водопадная диаграмма).

            Расширения: 
                5a. SHAP-модуль вернул ошибку → Backend возвращает 500, Frontend показывает «Не удалось рассчитать факторы». 
                7a. Нет факторов (все shap_value≈0) → отображается сообщение «Ни один фактор значимо не влияет».

        3.2.3 Функциональные требования
            ФТ-3.2.1 (Запрос факторов): 
                Система должна реализовывать API-эндпоинт GET /api/predictions/{prediction_id}/factors, 
                который возвращает список факторов риска для указанного прогноза.

            ФТ-3.2.2 (Отображение SHAP-значений): 
                В ответе на запрос факторов должны быть включены: название признака (feature_name), 
                его значение (feature_value), SHAP-значение (shap_value) и ранг влияния (rank).

            ФТ-3.2.3 (Хранение факторов): 
                Система должна сохранять рассчитанные факторы риска в таблице risk_factors БД, 
                связывая их с соответствующим прогнозом (prediction_id).

    3.3 Управление историей прогнозов

        3.3.1 Описание
            Пользователь может просмотреть историю всех ранее сделанных прогнозов, выполнить поиск по ФИО пациента, 
            удалить устаревшие записи и отсортировать их.

        3.3.2 Use Case
            Вариант использования: Просмотр истории выполненных прогнозов с поиском по ФИО

            Контекст использования: Аналитику нужно найти ранее сделанный прогноз для конкретного пациента.

            Область действия: Подсистема «История» (Frontend + Backend + БД)

            Уровень: Цель пользователя

            Основное действующее лицо: Страховой аналитик

            Участники: Frontend, Backend API, База данных

            User story: «Как аналитик, я хочу видеть список всех прогнозов с возможностью поиска по фамилии,
            чтобы быстро возвращаться к предыдущим расчётам».

            Предусловие: В БД есть хотя бы одна сохранённая запись прогноза.

            Минимальные гарантии успеха: При отсутствии записей отображается пустая таблица с сообщением «Нет данных».

            Гарантии успеха: Пользователь видит таблицу с колонками (ID, ФИО, возраст, прогноз, дата). 
            При вводе текста в поиск таблица фильтруется по ФИО.

            Триггер: Аналитик переходит в раздел «История расчетов» (навигационное меню).

            Описание (основной сценарий): 
                1. Аналитик открывает раздел «История». 
                2. Frontend отправляет GET /api/history (с параметрами page=1, limit=20). 
                3. Backend запрашивает данные из БД (таблица predictions). 
                4. БД возвращает записи. 
                5. Backend возвращает JSON-массив. 
                6. Frontend отображает таблицу. 
                7. Аналитик вводит текст в поле поиска (например, «Иванов») и нажимает Enter. 
                8. Frontend отправляет GET /api/history?search=Иванов. 
                9. Backend выполняет WHERE full_name LIKE '%Иванов%'. 
                10. Отфильтрованные записи возвращаются и отображаются.

            Расширения: 
                4a. Нет записей в БД → Backend возвращает пустой массив, Frontend показывает «История пуста». 
                8a. Поиск не дал результатов → отображается «Ничего не найдено».

        3.3.3 Функциональные требования
            ФТ-3.3.1 (Получение истории): 
                Система должна предоставлять API-эндпоинт GET /api/history, 
                который возвращает список всех прогнозов в формате HistoryResponse с поддержкой пагинации и поиска по параметру search.

            ФТ-3.3.2 (Отображение истории): 
                Frontend должен отображать историю расчетов в виде таблицы с колонками: 
                ID, ФИО пациента, возраст, прогнозируемая стоимость, дата создания, действия (удалить).

            ФТ-3.3.3 (Удаление прогноза): 
                Система должна предоставлять API-эндпоинт DELETE /api/history/{prediction_id}, 
                который удаляет указанный прогноз и все связанные с ним данные (каскадное удаление).

            ФТ-3.3.4 (Поиск в истории): 
                Интерфейс истории должен содержать поле поиска, 
                позволяющее фильтровать записи по вхождению подстроки в ФИО пациента.

    3.4 Удаление прогноза из истории

        3.4.1 Описание
            Пользователь может удалить выбранный прогноз

        3.4.2 Use Case
            Вариант использования: Удаление ошибочного или тестового прогноза

            Контекст использования: Аналитик заметил, что прогноз был сделан с неверными данными, и хочет очистить историю.

            Область действия: Подсистема «История» + БД

            Уровень: Подфункция

            Основное действующее лицо: Страховой аналитик

            Участники: Frontend, Backend API, База данных

            User story: «Как аналитик, я хочу удалять прогнозы, которые были сделаны по ошибке или для тестирования, 
            чтобы история оставалась релевантной».

            Предусловие: Пользователь находится в разделе «История», таблица отображает записи.

            Минимальные гарантии успеха: При отмене удаления запись остаётся нетронутой.

            Гарантии успеха: Выбранный прогноз и все связанные с ним факторы риска (каскадно) удаляются из БД. 
            Пользователь видит подтверждение и таблица обновляется.

            Триггер: Аналитик нажимает кнопку "удалить" в строке таблицы истории.

            Описание (основной сценарий): 
                1. В таблице истории аналитик видит строку с прогнозом. 
                2. Нажимает кнопку удаления (корзина) в этой строке. 
                3. Frontend показывает модальное окно: «Удалить прогноз для пациента [ФИО]?» с кнопками [Да] / [Нет]. 
                4. Аналитик нажимает [Да]. 
                5. Frontend отправляет DELETE /api/history/{prediction_id}. 
                6. Backend удаляет запись из таблицы predictions (каскадно удаляются risk_factors). 
                7. БД подтверждает удаление. 
                8. Backend возвращает статус 200 OK. 
                9. Frontend показывает уведомление «Прогноз удалён» и удаляет строку из таблицы.
            
            Расширения: 
                4a. Аналитик нажимает [Нет] → модальное окно закрывается, удаление не происходит. 
                6a. Запись с таким prediction_id не найдена → Backend возвращает 404, Frontend показывает «Прогноз уже удалён или не существует».

        3.4.3 Функциональные требования
            ФТ-3.4.1 (Отображение кнопки удаления): 
                В таблице истории прогнозов каждая строка должна содержать кнопку для удаления соответствующей записи.

            ФТ-3.4.2 (Подтверждение удаления): 
                При нажатии на кнопку удаления Frontend должен отображать модальное окно с текстом: 
                «Удалить прогноз для пациента [ФИО пациента]?» и кнопками [Да] и [Нет].

            ФТ-3.4.3 (Отправка запроса на удаление): 
                Если пользователь нажимает [Да], Frontend должен отправить HTTP-запрос 
                DELETE /api/history/{prediction_id} на Backend, где {prediction_id} — идентификатор выбранного прогноза.

    3.5 Экспорт отчета (PDF)

        3.5.1 Описание
            Пользователь может сформировать и скачать официальный отчет по результатам прогноза в формате PDF.

        3.5.2 Use Case
            Вариант использования: Формирование и скачивание официального отчёта по прогнозу

            Контекст использования: Аналитику нужно предоставить клиенту или руководству документ с результатами расчёта.

            Область действия: Подсистема «Отчёты» (Frontend + Backend + генератор PDF)

            Уровень: Цель пользователя

            Основное действующее лицо: Страховой аналитик

            Участники: Frontend, Backend API, PDF-генератор

            User story: «Как аналитик, я хочу скачать отчёт в PDF, содержащий все данные пациента, 
            прогноз и ключевые факторы риска, чтобы отправить его заинтересованным лицам».

            Предусловие: Прогноз с prediction_id существует (текущий или из истории).

            Минимальные гарантии успеха: Если генерация PDF не удалась, пользователь получает сообщение об ошибке, данные не теряются.

            Гарантии успеха: Пользователь получает PDF-файл с именем прогноз_{фамилия}_{дата}.pdf, который содержит: 
            ФИО, дату расчёта, прогнозируемую сумму, топ факторов риска, логотип компании (если есть).

            Триггер: Аналитик нажимает кнопку «Скачать отчёт в PDF» на экране результата или в строке таблицы истории.

            Описание (основной сценарий): 
                1. Аналитик находится на экране просмотра прогноза (текущий или из истории). 
                2. Нажимает кнопку «Экспорт в PDF». 
                3. Frontend отправляет POST /api/export/pdf с prediction_id в теле (или GET с параметром). 
                4. Backend получает запрос, запрашивает из БД данные прогноза и связанные факторы риска. 
                5. Backend генерирует PDF (библиотека ReportLab / WeasyPrint / др.). 
                6. Backend устанавливает Content-Type: application/pdf и заголовок Content-Disposition: attachment; filename=... 
                7. Frontend (браузер) инициирует скачивание файла. 8. Аналитик открывает PDF в просмотрщике.

            Расширения: 
                4a. Не найдены факторы риска → в PDF включается раздел «Факторы риска не рассчитаны» или они вычисляются на лету (SHAP). 
                5a. Ошибка генерации PDF → Backend возвращает 500, Frontend показывает «Не удалось создать отчёт». 
            

        3.5.3 Функциональные требования
            ФТ-3.5.1 (Генерация отчета): 
                Backend должен предоставлять API-эндпоинт для генерации PDF-отчета по prediction_id. 
                Это может быть реализовано через отдельный роутер prediction_pdf_router.

            ФТ-3.5.2 (Содержимое отчета): 
                PDF-отчет должен содержать: ФИО пациента, дату расчета, 
                прогнозируемую сумму расходов и список ключевых факторов риска с их влиянием.

            ФТ-3.5.3 (Инициация скачивания): 
                В пользовательском интерфейсе должна быть кнопка для скачивания отчета, 
                которая отправляет запрос на Backend и обрабатывает полученный PDF-файл.

    3.6 Сортировка записей в истории

        3.6.1 Описание
            Аналитик может кликнуть на заголовок столбца в таблице истории, чтобы отсортировать записи 
            (например, показать сначала самые дорогие прогнозы или самые свежие).

        3.6.2 Use Case
            Вариант использования: Сортировка прогнозов по выбранному столбцу (первый клик — по возрастанию)

            Контекст использования: Аналитик хочет упорядочить записи в истории, например, от самых дешёвых прогнозов 
            к самым дорогим или от старых к новым.

            Область действия: Подсистема «История» (только Frontend, клиентская сортировка)

            Уровень: Подфункция

            Основное действующее лицо: Страховой аналитик

            Участники: Веб-интерфейс (Frontend)

            User story: «Как аналитик, я хочу кликнуть на заголовок столбца и сразу увидеть записи, 
            отсортированные по возрастанию (от меньшего к большему), чтобы быстро находить наименьшие или ранние значения».

            Предусловие: Таблица истории уже загружена и отображает массив прогнозов. 
            Сортировка по данному столбцу ранее не применялась (или применялась по другому столбцу).

            Минимальные гарантии успеха: При ошибке сортировки таблица остаётся в исходном состоянии, данные не теряются.

            Гарантии успеха: Массив данных на клиенте сортируется по выбранному столбцу в порядке возрастания, таблица перерисовывается, 
            рядом с заголовком появляется кнопка сортировки.

            Триггер: Аналитик нажимает на заголовок столбца, поддерживающего сортировку (например, «Прогноз», «Дата» или «Возраст»).
            
            Описание (основной сценарий):
                1.Аналитик находится в разделе «История», видит таблицу с записями.
                2.Аналитик нажимает на заголовок столбца «Прогноз».
                3.Frontend определяет, что это первый клик по данному столбцу (текущее состояние сортировки 
                отсутствует или активна сортировка по другому полю).
                4.Frontend сортирует локальный массив данных по полю predicted_cost (created_at) 
                в порядке возрастания (числа — от меньшего к большему, даты — от старых к новым).
                5.Frontend устанавливает визуальную индикацию: рядом с заголовком отображается кнопка сортировки.
                6.Таблица перерисовывается, показывая записи в новом порядке.
            

        3.6.3 Функциональные требования
            ФТ-3.6.1 (Кликабельные заголовки столбцов): 
                Таблица истории прогнозов должна иметь заголовки столбцов: «Дата», «Пациент», «Возраст», «Прогноз», «Действия». 
                Заголовки столбцов, по которым возможна сортировка (дата, прогноз, возраст), должны быть кликабельными.

            ФТ-3.6.2 (Сортировка по возрастанию при первом клике): 
                При первом клике пользователя на заголовок столбца (например, «Прогноз») Frontend должен отсортировать 
                уже загруженный массив данных по соответствующему полю в порядке возрастания (от меньшего к большему для чисел, 
                от старых к новым для дат).

            ФТ-3.6.3 (Визуальная индикация направления сортировки): 
                После применения сортировки рядом с заголовком столбца должны отображаться две иконки:
                1.сортировка по возрастанию
                2.сортировка по убыванию 

4. Требования к данным
    4.1 Логическая модель данных
        Система оперирует четыремя основными сущностями: Patient, Synthetic_cohort, PredictionRecord и RiskFactor.

        Patient (Пациент) — содержит персональные данные пациента и не зависит от прогнозов.
        Атрибуты: id (PK), full_name, snils, phone, address.

        PredictionRecord (Запись прогноза) — хранит результат ML-предсказания и все входные параметры для этого прогноза.
        Атрибуты: id (PK), patient_id (FK к Patient.id), full_name, age, bmi, predicted_cost, created_at и др.

        RiskFactor (Фактор риска) — хранит детальный SHAP-анализ для каждого прогноза.
        Атрибуты: id (PK), prediction_id (FK к PredictionRecord.id), feature_name, feature_value, shap_value, rank.


    4.2 Получение, целостность и утилизация данных
        Получение данных: 
            Backend получает данные о пациенте и прогнозе через API-запросы. 
            Эти данные проходят валидацию на основе Pydantic-схем (например, PredictionInput).

        Целостность данных: 
            Каскадное удаление: при удалении записи из таблицы predictions, все связанные с ней 
            записи из таблицы risk_factors автоматически удаляются (cascade="all, delete-orphan").

        Утилизация данных: 
            Данные удаляются пользователем через интерфейс (кнопка "Удалить" в истории), 
            что приводит к вызову API DELETE /api/history/{prediction_id}.

5. Требования к внешним интерфейсам
    5.1 Пользовательские интерфейсы
        Форма ввода данных: Набор полей ввода, соответствующих PredictionInput, сгруппированных по логическим блокам.
        Область результатов: Отображает predicted_cost и кнопки для просмотра факторов риска и скачивания PDF.
        Страница истории: Таблица с прогнозами, поле поиска и кнопки удаления.

    5.2 Программные интерфейсы (API)
        Backend предоставляет REST API. Взаимодействие между Frontend и Backend происходит через HTTP-запросы. 
        Основные эндпоинты:
            POST /api/predict — Принимает данные пациента, возвращает prediction_id и predicted_cost.

            GET /api/predictions/{prediction_id}/factors — Возвращает список факторов риска.

            GET /api/history — Возвращает историю прогнозов.

            DELETE /api/history/{prediction_id} — Удаляет прогноз и связанные данные.

            POST /api/ocr/patient-form — Принимает файл (скан/фото), возвращает распознанные поля.

            GET /api/health — Эндпоинт для проверки работоспособности сервиса.

6. Атрибуты качества
    6.1 Производительность:
        Время отклика API (Latency): ≤ 2000 мс для 95% запросов на получение прогноза.

        Частота ошибок API: < 0.5% от общего количества запросов.

        Время начальной загрузки Frontend (Initial Load Time): ≤ 3000 мс для 95% сессий.

    6.2 Точность модели (ML-метрики):
        MAE (Mean Absolute Error) — основная метрика.

        MAPE (Mean Absolute Percentage Error) — дополнительная относительная метрика.

        RMSE (Root Mean Square Error) — метрика для контроля крупных ошибок.

        R² (коэффициент детерминации) — вспомогательная метрика.
